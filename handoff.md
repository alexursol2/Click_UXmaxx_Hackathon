## 🎯 TL;DR (по-русски)

**Nimbus** — это демо SaaS-подписки с «абстракцией блокчейна». Пользователь:

1. **Логинится по email** (без seed-фразы, без кошелька — встроенный кошелёк Magic).
2. Его обычный адрес (EOA) **превращается в Universal Account** через EIP-7702.
3. Платит **рекуррентную подписку** ($5/мес или 0.001 ETH) из **единого баланса**,
   который собирается автоматически с **любой сети** — деньги могут лежать на Base,
   а мерчант получает оплату на Arbitrum. **Без попапов, без ручного бриджинга,
   без переключения сетей.**

**Это уже работает на mainnet — не мокап.** ETH на Base оплатил мерчанта в USDC на
Arbitrum, автоматически сконвертировалось и забриджилось, всё подтверждено ончейн.

**Хакатон:** UXmaxx, трек **Universal Accounts**. Судейство: **40% UX, 30%
заметность использования Universal Accounts.** Проект бьёт ровно в эти два.

**Партнёрские технологии в проекте:** Magic (embedded wallet) + Particle Network
(Universal Accounts) + Arbitrum (сеть расчёта) + EIP-7702 (стандарт делегации).

## 🎬 Сценарий демо-видео (то, что показывать на экране)

Порядок кадров — от «боли» к «магии». Каждый пункт = кадр/сцена.

1. **Экран логина.** Показать: только поле email, анимированный лого-часы,
   подпись «No wallet, no seed phrase». → *Голос: «Обычный пользователь. Никаких
   кошельков, никаких seed-фраз — только почта».*
2. **Ввод email → OTP-код из письма → вход.** Кошелёк создаётся сам. → *«Под
   капотом Magic создал ему встроенный некастодиальный кошелёк».*
3. **Dashboard: большая цифра единого баланса** (например `$5.34`) с чипом
   **«Aggregated from N chains — spendable as one»** и разбивкой «где деньги
   реально лежат» (ETH · Base $3.11 / Arbitrum $1.15). → *«Вот его деньги —
   размазаны по разным сетям, но показаны как один баланс».* **Это ключевой кадр
   для критерия „заметность Universal Accounts" (30%).**
4. **Нажатие «Upgrade to Pro».** Появляется **пошаговый прогресс**: Sourcing →
   Converting across chains → Bridging → Paying merchant → Confirming, и
   **баннер-путешествие „Base → Arbitrum"** с бегущей точкой. → *«Один клик. Он
   не выбирает сеть, не бриджит вручную — Universal Account сам находит деньги на
   Base, конвертирует, бриджит на Arbitrum и платит мерчанту».* **Это „the shot"
   для видео — крупный план на анимации Base → Arbitrum.**
5. **«Payment confirmed» + «Pro active».** Показать, что это подтверждено ончейн
   (не просто UI). → *«И — ни одного попапа за всё время. Подпись прошла headless».*
6. **Рекуррентность:** таймер «Next billing cycle: 10s» и авто-списание. Показать,
   как проходят 2–3 автоматических списания подряд **без единого клика и попапа**. →
   *«Настоящая рекуррентная подписка. Каждое списание — реальная ончейн-транзакция».*
7. **Billing history** внизу: реальные ончейн-чеки со статусами Settled. → *«Все
   платежи — публичные, проверяемые транзакции. База данных подписки — это сам
   блокчейн».*
8. **(Опционально, сильный финал) Reload страницы** — «Pro active» остаётся, потому
   что статус выводится из ончейн-истории. → *«Бэкенда нет. Состояние подписки
   живёт в блокчейне».*

**Слоган-кандидаты:** «One balance. One click. Any chain.» / «Subscriptions,
without the crypto.» / «Email in. Chain-abstracted out.»

## 🧩 Что это, одним абзацем (для слайда «Problem/Solution»)

**Проблема:** крипто-подписки невозможны для обычных людей — нужен кошелёк,
seed-фраза, газовый токен в правильной сети, ручной бриджинг, и попап на каждое
списание (а значит рекуррентность в принципе не работает без открытого кошелька).
**Решение:** email-логин + Universal Account + EIP-7702 = подписка, которая
ощущается как обычный SaaS (Netflix-подобный UX), но работает на любой сети из
единого баланса и без попапов.

## 🏗 Как это работает (для слайда «Architecture»)

```
 ┌──────────────┐   email OTP    ┌──────────────────┐    EIP-7702    ┌─────────────────────┐
 │  Magic wallet │ ─────────────▶ │ useUniversal-     │ ─────────────▶ │ Particle Universal   │
 │  (EOA, no seed│  headless sign │ Upgrade()         │   type-4 tx    │ Account: единый      │
 │  phrase)      │                │ = «продукт»       │                │ кросс-чейн баланс    │
 └──────────────┘                └──────────────────┘                └─────────────────────┘
```

- **Magic** превращает email во встроенный кошелёк. Подпись **headless** (без
  попапов) — именно поэтому рекуррентные списания вообще возможны. Также Magic
  подписывает EIP-7702-авторизацию (`sign7702Authorization()`).
- **Particle Universal Accounts (SDK v2)** абстрагирует балансы со всех сетей в
  одну цифру и одно место, откуда можно тратить.
- **EIP-7702** превращает обычный email-EOA в этот Universal Account. Авторизация
  «подкладывается» в первую транзакцию — отдельной setup-транзакции нет.

### Поток списания (важно для точности в презентации)

На бэкенде Particle v2 обычный `transfer` тратит только то, что **уже лежит на
сети расчёта**. Кросс-чейн источник — это **Convert** (`createConvertTransaction`),
который сперва перекидывает деньги на нужную сеть. Один клик «Upgrade» прячет за
собой две транзакции и два ожидания подтверждения:

```
transfer ── деньги уже на сети расчёта? ──▶ оплачено ✓
   │ нет
   ▼
convert (любая сеть → сеть расчёта) ──▶ ждём подтверждения ончейн
   ▼                                        │ transfer упал здесь?
transfer ──▶ ждём подтверждения ончейн      ▼
   ▼                                  PARTIAL: деньги перемещены, мерчант не оплачен.
«Pro active»                          Повтор ПРОДОЛЖАЕТ с transfer —
                                      никогда не конвертит и не платит дважды.
```

## 💡 «Почему это технически сложно» (тезисы для судей / Q&A)

Абстракция блокчейна выглядит как одна цифра и одна кнопка. Сложность — во всём,
что под ними, и что наивная интеграция делает неправильно. Проект уже «оплатил»
эти уроки на mainnet:

1. **Подтверждение ≠ отправка.** `sendTransaction` вернулся — значит операция
   *принята*, а не *исполнена*. Кросс-чейн платёж расчётывается асинхронно; всё,
   что зависит от прихода денег, поллит статус до терминального (7 = FINISHED).
   UI показывает «Pro» только после статуса 7, никогда раньше.
2. **Transfer тратит только сеть расчёта.** Поэтому «заплатить $5 USDC на Arbitrum
   с ETH на Base» — это Convert → wait → transfer → wait, а не один вызов.
3. **Кросс-чейн платёж может «пол-исполниться».** Если Convert прошёл, а перевод
   упал — деньги уже на сети расчёта, а мерчант не оплачен. Приложение это
   ловит, честно показывает, и **повтор продолжает с перевода** — не конвертит и
   не платит дважды.
4. **Буфер — это догадка; квота — истина.** Перед Convert приложение спрашивает
   у SDK, сколько *реально придёт*, и доливает до цены + газ. Фиксированный буфер
   только «пол».
5. **7702-делегация — по-сетевая.** Делегация на Base не делегирует на Arbitrum;
   каждой сети нужна своя авторизация, подкладываемая в первую транзакцию.
6. **Mainnet-бэкенды уходят в maintenance.** Реальная инфра отдаёт `-32801`,
   `-32653`, REFUND_FAILED. В демо нельзя показывать сырой JSON-RPC код — они
   мапятся в спокойный текст, транзиентные — с одним авто-повтором.
7. **Рекуррентность на mainnet жжёт реальные деньги.** Guard по низкому балансу
   ставит авто-списание на паузу до опустошения кошелька; UI показывает остаток
   циклов и живой баланс, чтобы демо не умерло на середине питча.

## ▶️ Как запустить (для скринкаста)

```bash
cd D:\Other\Hackathons\uxmaxx-universal-subscriptions
npm install            # если ещё не стоял
npm run dev            # → http://localhost:3000
```

Ключи уже в `.env.local` (Magic + Particle + merchant). Демо-режим включён:
списание каждые **10 секунд, 3 раза** за сессию (`NEXT_PUBLIC_BILLING_INTERVAL_SECONDS=10`,
`NEXT_PUBLIC_BILLING_MAX_CYCLES=3`). Текущая тестовая цена в `.env.local` —
`NEXT_PUBLIC_SUBSCRIPTION_PRICE_USD` (для боевого демо можно вернуть 5).

**Важно для записи:** каждое списание — реальные деньги на mainnet. Перед записью
проверь баланс owner-кошелька скриптом:
`node --env-file=.env.local scripts/check-account.mjs 0xАДРЕС`, и посчитай стоимость
прогона: `цена × 3` (возвращается на твой же merchant) + комиссия за кросс-чейн-цикл
(`node --env-file=.env.local scripts/fees.mjs 0xАДРЕС 0.5`).

## 🔒 Безопасность «zero popups» (частый вопрос судей — заготовь ответ)

Headless-подпись — это не «обход согласия», а осознанный дизайн:
- Ключ **никогда не покидает изолированную среду Magic**; приложение держит только
  EIP-1193 провайдер, не сырой ключ.
- Пользователь **один раз явно авторизует делегацию** (клик «Upgrade to Pro» = его
  согласие), и 7702-авторизация ограничена имплементацией Universal Account.
- Каждое списание — **фиксированная сумма фиксированному мерчанту**; ничего
  произвольного не подписывается.
- Это ровно модель доверия любой «card-on-file» подписки: авторизовал раз —
  списывается без повторных запросов. Только тут сумма и реестр **публичны и
  проверяемы ончейн**.

## ⚠️ Известные ограничения / что дальше (честно — для слайда «Roadmap»)

- **Полностью автономный биллинг** пока требует открытой сессии пользователя
  (Magic подписывает на клиенте). Настоящий merchant-initiated биллинг без
  пользователя — это **session keys** с лимитом трат и мерчантом (главный next step).
- **Pay-with селектор** (выбрать, каким активом/сетью платить) — не сделан, в скоупе
  дальше.
- **Сети расчёта** — пока только Arbitrum One (7702-режим Particle поддерживает
  Ethereum, Base, BNB, Arbitrum, XLayer + адрес на Solana).
- Мелочи: в корне мог остаться `build-check.log` (безвредно, в .gitignore) —
  удалить `rm build-check.log`.
